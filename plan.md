# Pomodoro Timer 段階実装プラン

## 概要

このプランは、`architecture.md` と `features.md`、および UI モック画像をもとに、Pomodoro タイマーアプリを段階的に実装するための手順を整理したものである。

UI モックから読み取れる初期画面の主要要素は次のとおりである。

- タイトル付きの単一カードレイアウト
- 現在モード表示
- 円形プログレス付きの残り時間表示
- 開始ボタン
- リセットボタン
- 今日の進捗カード

このため、実装は「まず画面を成立させる」「次にブラウザ内でタイマーを完成させる」「最後に Flask API と永続化を接続する」という順序で進める。

## 実装方針

- 各ステップで必ず動作確認可能な状態を作る
- 先に UI の骨組みと操作感を作り、その後にバックエンド連携を追加する
- ドメインロジックは早めに純粋関数として分離し、後続の API 実装で再利用する
- 永続化は UI とタイマーの動作が安定してから追加する
- テストは後付けではなく、ドメインロジックとサービス層から段階的に追加する

## ステップ別実装計画

### Step 1. プロジェクト骨組みの作成

目的:
Flask アプリ、テンプレート、静的ファイル、ドメイン、サービス、リポジトリ、テストの基本構成を作る。

実装内容:
- `1.pomodoro/app.py` に Flask アプリのエントリポイントを作成する
- `templates/`、`static/css/`、`static/js/`、`domain/`、`services/`、`repositories/`、`tests/` を作成する
- `GET /` のみを持つ最小構成で画面表示を可能にする
- 将来の分割に備えて JS と Python ファイルを空で用意する

完了条件:
- Flask を起動するとトップ画面が表示される
- 推奨ディレクトリ構成が一通り揃っている

### Step 2. モック準拠の静的 UI を作る

目的:
添付画像に近い見た目の UI を、まだロジックなしで表示できる状態にする。

実装内容:
- タイトル、モード表示、タイマー表示、円形プログレス、開始ボタン、リセットボタン、今日の進捗カードを HTML に配置する
- CSS 変数で色、余白、角丸、影を定義する
- 円形プログレスを SVG `circle` で描画する
- デスクトップとモバイルの両方でレイアウトが破綻しないように調整する
- 初期表示値として `25:00`、完了数、集中時間のモック値を表示する

完了条件:
- 画像の主要レイアウトを再現した画面が表示される
- 画面がレスポンシブに崩れず表示される

### Step 3. フロントエンドだけで動くタイマー MVP を作る

目的:
バックエンド未接続の状態でも、開始・一時停止・再開・リセットが動くタイマーを完成させる。

実装内容:
- `timer-engine.js` に `idle / running / paused / finished` の状態遷移を実装する
- `Date.now()` 差分ベースで残り時間を計算する
- 開始、一時停止、再開、リセットのイベントを実装する
- タイマー数字表示と円形プログレスをリアルタイム更新する
- 作業モード終了時に `finished` 状態へ遷移させる

完了条件:
- 画面上の操作だけで 1 セッションのタイマーが正しく進行する
- 一時停止や再開で時間ずれが目立たない

### Step 4. フロントエンド状態管理と localStorage 復元を追加する

目的:
ブラウザ再読み込み後も、Pomodoro の状態を復元できるようにする。

実装内容:
- `pomodoro-state.js` に画面状態の保存ロジックを実装する
- 現在モード、残り時間、セッション状態、完了回数、集中時間を管理する
- localStorage へ状態を保存する
- 初期ロード時に localStorage から状態を復元する
- 保存データが壊れている場合のフォールバックを実装する

完了条件:
- ページを再読み込みしてもタイマー状態と進捗表示が復元される
- 不正データで画面が壊れない

### Step 5. ドメインロジックを純粋関数として実装する

目的:
モード遷移と集計ロジックを UI や Flask から独立させる。

実装内容:
- `domain/pomodoro_rules.py` に次モード判定関数を実装する
- 長休憩判定関数を実装する
- 日次集計更新関数を実装する
- 初期表示用データ組み立て関数を実装する
- Python 側 Clock 抽象化の土台を用意する

完了条件:
- ドメインロジックが副作用なしで呼び出せる
- 後続のサービス層から再利用できる形になっている

### Step 6. ドメインロジックの単体テストを先に作る

目的:
以降のバックエンド実装の基準になるルールを、先にテストで固定する。

実装内容:
- `test_pomodoro_rules.py` を作成する
- 作業終了後に短休憩へ進むケースを検証する
- 規定回数到達時に長休憩へ進むケースを検証する
- 日次集計更新結果を検証する
- 初期表示データの生成結果を検証する

完了条件:
- ドメインルールの主要分岐がテストでカバーされる

### Step 7. Flask API の器を作る

目的:
フロントエンドが将来接続する API の入口だけを先に揃える。

実装内容:
- `GET /api/state` を追加する
- `GET /api/stats/today` を追加する
- `POST /api/settings` を追加する
- `POST /api/session/complete` を追加する
- `POST /api/session/reset` を追加する
- いったんは固定値または In-memory データで応答する

完了条件:
- 必要な API パスがすべて応答する
- フロントエンド側から疎通確認できる

### Step 8. リポジトリとサービス層を In-memory 実装でつなぐ

目的:
API の固定値返却をやめて、責務分離されたバックエンド構造へ移行する。

実装内容:
- `settings_repository.py` に設定保存・取得の interface と In-memory 実装を作る
- `stats_repository.py` に統計保存・取得、セッション記録の interface と In-memory 実装を作る
- `settings_service.py` に設定保存・取得ロジックを実装する
- `stats_service.py` に統計更新、セッション完了、リセット処理を実装する
- Clock 抽象化をサービス層から利用する

完了条件:
- API がサービス経由で一貫してデータを返す
- データアクセスが routes に直書きされていない

### Step 9. フロントエンドを Flask API 連携に切り替える

目的:
初期表示、完了記録、統計更新をバックエンド基準にする。

実装内容:
- `pomodoro-api.js` に API 通信処理を実装する
- 初期表示時に `/api/state` を取得する
- セッション完了時に `/api/session/complete` を呼び出す
- リセット時に `/api/session/reset` を呼び出す
- 今日の進捗カードを API 応答に基づいて更新する
- フロントエンド側で重複していた集計ロジックを削減する

完了条件:
- 表示中の完了数と集中時間が API の返却値に一致する
- セッション完了後に最新統計が画面へ反映される

### Step 10. 設定機能を実装する

目的:
作業時間や休憩時間を変更できるようにし、ルールを可変にする。

実装内容:
- `work_minutes`、`short_break_minutes`、`long_break_minutes`、`cycles_before_long_break` を扱う設定モデルを実装する
- バックエンドで設定の保存と取得を実装する
- UI 上で設定変更の入口を追加する
- 設定変更後にタイマー初期値とモード遷移へ反映する
- 設定値のバリデーションを追加する

完了条件:
- 設定変更が保存され、再読み込み後も保持される
- タイマー初期値と長休憩判定が設定値に従う

### Step 11. 永続化を SQLite または JSON ストレージへ置き換える

目的:
アプリ再起動後も設定、統計、履歴が保持されるようにする。

実装内容:
- `settings`、`daily_stats`、`session_log` の保存形式を実装する
- In-memory Repository を本番用 Repository に差し替える
- 日付単位の統計取得を実装する
- セッション履歴の保存を実装する
- 必要なら初期データ生成処理を追加する

完了条件:
- サーバー再起動後も設定と統計が保持される
- セッション完了履歴が保存される

### Step 12. バックエンドテストを充実させる

目的:
サービス層と API の正しさを担保する。

実装内容:
- `test_stats_service.py` で集計更新、完了記録、リセットを検証する
- `test_api.py` で `/api/state`、`/api/settings`、`/api/session/complete`、`/api/session/reset`、`/api/stats/today` を検証する
- In-memory Repository を利用してテストを高速に保つ
- 異常系入力のテストを追加する

完了条件:
- 主要 API とサービスの正常系・異常系がテストで確認できる

### Step 13. 仕上げと UX 調整

目的:
MVP として自然に使える完成度まで整える。

実装内容:
- ボタンラベルやモード文言を整える
- ローディングやエラー表示を追加する
- API 通信失敗時のフォールバックを入れる
- 画面再描画のちらつきを抑える
- 不要な責務混在がないか確認する

完了条件:
- MVP として一通りの操作が安定して行える
- UI、フロントエンド、バックエンドの責務分離が保たれている

## 実装順の理由

この順番にする理由は次のとおりである。

- 最初に静的 UI を作ることで、モックとの差分を早期に確認できる
- タイマーの中心課題はフロントエンドの時間計算なので、先にブラウザ側で完成させる方が安全である
- モード遷移や集計はドメインロジックとして先に固定した方が、API 実装のぶれを防げる
- 永続化は最後に差し替えればよく、序盤に入れると開発速度を落としやすい

## 1 ステップごとの推奨成果物

- Step 1: Flask でトップ画面が表示される
- Step 2: モック準拠の UI が表示される
- Step 3: ブラウザだけでタイマーが動く
- Step 4: 再読み込み後も状態が復元される
- Step 5-6: ドメインルールとそのテストが成立する
- Step 7-9: API とフロントエンドが接続される
- Step 10-11: 設定変更と永続化が成立する
- Step 12-13: テストと UX を仕上げる
